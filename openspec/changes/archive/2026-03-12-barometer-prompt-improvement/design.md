## Context

晴雨表功能透過 `SYSTEM_PROMPT` + `buildUserMessage` 組合提供 LLM 分析所需的全部資訊。目前問題集中在兩處：

1. **期貨部位日變化的語意歧義**：`buildUserMessage` 直接輸出原始數字（如「日變化：-300」），當底倉為負值（淨空方向）時，-300 代表空方加碼，但模型容易解讀為「減少 300 口」，導致方向性誤判。
2. **system prompt 範例失真**：規格要求 summary 250-350 字，但兩個示範範例均約 130 字，模型依範例學習，實際輸出偏短且資訊量不足。

## Goals / Non-Goals

**Goals:**
- 消除期貨淨未平倉日變化值的語意歧義，讓模型無需自行推斷方向
- 讓 system prompt 的範例與 summary 字數規格一致
- 壓縮 system prompt 中的冗餘內容，降低 token 消耗

**Non-Goals:**
- 不改動 API 介面或資料庫 schema
- 不換 LLM 模型
- 不修改前端呈現

## Decisions

### 決策一：在 buildUserMessage 加入方向語意標籤

**做法**：對以下欄位的日變化值，根據底倉方向輸出說明文字：
- `finiTxfNetOiChange`（配合 `finiTxfNetOi`）
- 散戶微台、小台（配合 `retailTmfNetOi`、`retailMxfNetOi`）

轉換邏輯：

```
底倉為正（淨多方向）：
  變化 > 0 → 「多單增加 N 口」
  變化 < 0 → 「多單減少 N 口」
  變化 = 0 → 「持平」

底倉為負（淨空方向）：
  變化 < 0 → 「空單增加 N 口」
  變化 > 0 → 「空單減少 N 口」
  變化 = 0 → 「持平」

底倉為零：
  變化 > 0 → 「新增多單 N 口」
  變化 < 0 → 「新增空單 N 口」
```

**替代方案考慮**：在 system prompt 加說明「負底倉時負變化代表空方加碼」→ 不採用，因為語意說明要求模型自行計算，不如直接給結果可靠。

### 決策二：Summary 字數規範調整為彈性 200-350 字

**做法**：規格從固定「250-350 字」改為彈性「200-350 字」，同步更新 system prompt 與範例。

**理由**：
- 固定上下限會迫使模型在訊號高度一致時填充冗餘內容
- 彈性範圍讓模型在訊號複雜時自然寫到 300+ 字，訊號一致時 200 字說清楚即可
- 範例以 250-300 字示範「中等複雜」情境，提供實際錨點
- 前端 `.summary-text` 無截斷，最長 350 字仍在可接受的 Hero Card 高度內

### 決策三：壓縮 system prompt 的基礎指標說明

**做法**：移除「正值=買超，負值=賣超」等模型早已知曉的基礎定義，只保留：
- 反向指標邏輯（散戶是反向指標）
- 非直覺規則（txoPutCallRatio 極端值、「翻轉才是強訊號」）
- 指標優先順序（微台 > 小台）

## Risks / Trade-offs

- **方向標籤在底倉接近零時較難處理**：底倉 ±100 口左右的「接近零」情境，方向語意標籤可能不夠精確 → 接受此邊際狀況，不影響主要情境
- **壓縮 prompt 後可能遺漏某些規則**：需要在修改後觀察幾個實際交易日的輸出品質 → 保留一週觀察期，若品質下降可快速還原
